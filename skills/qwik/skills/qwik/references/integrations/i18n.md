# Integration: Internationalization (i18n)

> Source: `packages/docs/src/routes/docs/integrations/i18n/index.mdx`

## Approaches

| Approach | Pros | Cons |
| ---------- | ------ | ------ |
| **Runtime** | Simple, no build step | Eager-loads all translations; runtime cost |
| **Compile-time** | Zero runtime cost; lazy-loaded strings | Extra build step; language change requires reload |

**Qwik recommendation:** compile-time on client (lazy-loading), runtime on server (single deployment for all locales).

## Recommended libraries

### Paraglide JS (compile-time)

```bash
# Follow official Vite guide:
# https://inlang.com/m/gerre34r/library-inlang-paraglideJs/vite
```

```tsx
import * as m from './src/paraglide/messages';
// m.hello({ name: 'World' }) → localized string
```

- Generates type-safe translation functions.
- ICU message format.
- Only bundles used translations.

### compiled-i18n (compile-time + runtime)

```bash
npx qwik add compiled-i18n
```

```tsx
import { _ } from 'compiled-i18n';
const msg = _`Hello ${name}`;
```

- Supports both compile-time (inlined) and runtime (map lookup) modes.
- Auto-extracts new keys during build.
- Deepl auto-translation via `deepl-localize`.

### qwik-speak (runtime)

```bash
# Follow: https://github.com/robisim74/qwik-speak/blob/main/docs/quick-start.md
```

- Dynamic language switching without reload.
- Deep Qwik reactivity integration.
- Supports dates, numbers, pluralization, ICU.

## Key points

- Qwik does not provide a built-in i18n solution — it exposes low-level APIs for libraries.
- Translation maps loaded at runtime prevent Qwik's lazy loading from working optimally.
- Compile-time approaches align best with Qwik's streaming / lazy-loading model.
